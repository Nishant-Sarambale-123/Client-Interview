> ✅ **“Liveness probe failing in EKS – how do you troubleshoot?”**

You can present this in a **step-by-step layered approach** 👇

***

# ✅ 🔷 High-Level Answer (Start with this)

> “If a liveness probe fails in EKS, I follow a structured approach starting from Kubernetes events, then pod/container health, probe configuration, application logs, and underlying infrastructure like nodes and networking.”

***

# ✅ 🔷 Step-by-Step Troubleshooting Approach

## 🟢 1. Check Pod Status & Events (First step ALWAYS)

```bash
kubectl describe pod <pod-name>
```

Look for:

* `Liveness probe failed`
* `Back-off restarting container`
* Restart count

👉 This tells:

* Is it really probe failure?
* What error is returned? (timeout / connection refused / 500)

***

## 🟢 2. Check Restart Count

```bash
kubectl get pod <pod-name>
```

👉 If restarts are high:

* Liveness probe is killing container repeatedly

***

## 🟢 3. Verify Probe Configuration

```bash
kubectl get pod <pod-name> -o yaml
```

Check:

```yaml
livenessProbe:
  httpGet:
    path: /health
    port: 8080
  initialDelaySeconds: 30
  periodSeconds: 10
  timeoutSeconds: 2
  failureThreshold: 3
```

### ✅ Common Issues:

* Wrong path (`/health`)
* Wrong port
* Timeout too low
* Initial delay too short → app not ready

***

## 🟢 4. Manually Test Probe Inside Pod

```bash
kubectl exec -it <pod-name> -- sh
```

Test:

```bash
curl localhost:8080/health
```

👉 Helps identify:

* App not responding
* Slow response
* Endpoint broken

***

## 🟢 5. Check Application Logs

```bash
kubectl logs <pod-name>
```

👉 Look for:

* Startup delays
* Crashes
* Dependency errors (DB connection, etc.)

***

## 🟢 6. Check Resource Issues

```bash
kubectl top pod
kubectl describe pod
```

👉 Look for:

* CPU throttling
* OOMKilled

### ✅ If found:

* Increase CPU/memory
* Optimize app

***

## 🟢 7. Check Node-Level Issues (EKS-specific)

```bash
kubectl describe node <node-name>
```

Check:

* Memory pressure
* Disk pressure
* Network issues

👉 In EKS:

* Node overload can delay response → probe fails

***

## 🟢 8. Check Networking / Service Issues

If probe is external:

* Network policies
* Security groups (EKS + VPC)
* Service misconfiguration

***

## 🟢 9. Compare Liveness vs Readiness

👉 Interview point (important):

| Probe     | Purpose                      |
| --------- | ---------------------------- |
| Liveness  | Restarts unhealthy container |
| Readiness | Controls traffic             |

⚠️ Common mistake:

* Using liveness probe for readiness → causes crash loops

***

## 🟢 10. Validate Fix with Better Config

Example improved config:

```yaml
livenessProbe:
  httpGet:
    path: /health
    port: 8080
  initialDelaySeconds: 60
  periodSeconds: 15
  timeoutSeconds: 5
  failureThreshold: 5
```

***

# ✅ 🔷 Root Cause Categories (Mention in Interview)

You can summarize like this:

👉 “Failures usually fall into these categories:”

1. **Application issues**
   * App crash
   * Health endpoint failure

2. **Configuration issues**
   * Wrong path/port
   * Timeout too strict

3. **Resource issues**
   * CPU throttling
   * Memory OOM

4. **Startup delay**
   * Needs higher initialDelay

5. **Infrastructure issues (EKS)**
   * Node pressure
   * Network latency

***

# ✅ 🔷 Pro Tip (Interview Impact 💥)

👉 Say this line:

> “I always test the health endpoint manually from inside the container and correlate it with probe timings to distinguish between app issues and configuration problems.”

***

# ✅ 🔷 Sample Final Answer (Short Version)

If interviewer asks for a quick answer:

> “I start by checking `kubectl describe pod` for liveness probe errors and restart counts. Then I verify probe configuration like path, port, and timing. I exec into the pod to manually test the health endpoint. Next, I check application logs and resource usage to see if CPU/memory or startup delays are causing failures. In EKS, I also validate node health and networking. Finally, I adjust probe parameters like `initialDelaySeconds` and `timeoutSeconds` to stabilize the application.”

***

✅ Avoid:
